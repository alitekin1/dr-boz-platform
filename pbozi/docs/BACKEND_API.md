# JGPTi Backend API Documentation

## 1. Overview
JGPTi is a robust backend system built with **FastAPI**, designed to power AI-driven interactions via multiple channels. Following a recent architectural shift, the UI has been removed, and the backend now serves as the primary interface for API clients and the Telegram bot.

### Key Architectural Pillars:
- **FastAPI**: Provides a high-performance, asynchronous web framework.
- **SQLAlchemy (Async)**: Manages database persistence with support for complex relationships and migrations.
- **RAG (Retrieval-Augmented Generation)**: Integrated document indexing (PDF/TXT/MD) and vector search capabilities.
- **Multi-Provider Support**: Seamlessly switch between different LLM providers (e.g., OpenAI, Google).
- **Usage & Credit System**: A sophisticated metering and wallet system for tracking and billing AI usage.

---

## 2. Authentication
The JGPTi API uses a simple yet secure authentication mechanism for administrative and sensitive operations.

### Admin Authentication
Admin-level routes require an **HTTP Bearer Token**. The token value must match the `ADMIN_PASSWORD` configured in the backend environment.

- **Header**: `Authorization: Bearer <ADMIN_PASSWORD>`
- **Scope**: Required for all routes under `/api/admin`.

### Public/User Routes
Routes under `/api/account` and certain `/api` endpoints are accessible based on user identifiers (e.g., Telegram User ID) or are public for basic status checks.

## 3. Data Models Overview
The system is built around several core entities defined in `models.py`:

| Model | Description |
| :--- | :--- |
| **UserPreference** | Profiles for users, including Telegram IDs, credit balances, and status. |
| **Wallet** | Handles the minor-unit currency (USD) balances and versioning for safe updates. |
| **Project** | Logical containers for related chats and documents. |
| **Chat** | Individual conversation threads linked to a project or standalone. |
| **Message** | Content within a chat (user, assistant, or system roles). |
| **Document** | Uploaded files processed and indexed for RAG. |
| **Provider & Model** | Configuration for LLM services and specific models with pricing. |
| **Tool & Binding** | Extensible tool system allowing models to call external functions. |
| **UsageEvent** | Granular logs of token usage, costs, and operation metadata. |
| **CreditLedgerEntry** | Audit trail for all financial adjustments and usage deductions. |

## 4. API Endpoints

### 4.1 Account Routes (`/api/account`)
Used for user onboarding and status checks.

- `GET /api/account/status/by-telegram/{telegram_user_id}`: Retrieve account status and onboarding progress.
- `GET /api/account/summary/{user_id}`: Get a detailed summary of a user's profile.
- `GET /api/account/normalize-phone`: Utility to preview how a phone number will be formatted.
- `GET /api/account/learning-preferences/{user_id}`: Get learning-preference onboarding status/profile.
- `GET /api/account/learning-preferences/by-telegram/{telegram_user_id}`: Same status lookup by Telegram ID.
- `POST /api/account/learning-preferences/by-telegram/{telegram_user_id}/start`: Start or resume conversational onboarding (`?restart=true` supported).
- `POST /api/account/learning-preferences/by-telegram/{telegram_user_id}/turn`: Submit one user answer turn.
- `POST /api/account/learning-preferences/by-telegram/{telegram_user_id}/skip`: Skip onboarding.
- `POST /api/account/learning-preferences/by-telegram/{telegram_user_id}/finalize`: Force final analysis/profile generation.

### 4.2 Main Routes (`/api`)
Core functionality for messaging, projects, and documents.

#### Projects & Documents
- `GET /api/projects`: List all projects.
- `POST /api/projects`: Create a new project.
- `POST /api/projects/{project_id}/documents`: Upload a document (Multipart).
- `POST /api/projects/{project_id}/documents/upload-base64`: Upload a document (Base64 encoded).
- `GET /api/generated-pdfs/{filename}`: Download a PDF generated by the builtin `pdf_generator` tool.

#### Chats & Messages
- `GET /api/chats`: List chats (filter by `project_id`).
- `POST /api/chats`: Initialize a new chat session.
- `POST /api/chats/{chat_id}/messages`: Send a message and receive an AI response (supports RAG and Tools).
- `GET /api/chats/{chat_id}/messages`: Retrieve conversation history.

`POST /api/chats/{chat_id}/messages` response now includes optional reasoning metadata for compatible reasoning models:
- `message.reasoning_summary`: short summary string suitable for a "Thinking about..." label.
- `message.reasoning_stages`: ordered list of stage objects (`index`, `topic`, `summary`) for UI topic transitions.
- `reasoning`: same structured object at top-level for convenience (`available`, `summary`, `stages`).

Message request supports optional user linkage:
- `user_id` or `telegram_user_id` can be included to bind a chat to a user profile.
- once linked, learning-preference personalization is injected automatically on future turns for that chat.

### 4.3 Admin Routes (`/api/admin`)
*Requires Admin Bearer Token.*

- `GET /api/admin/stats`: System-wide statistics (counts of users, messages, etc.).
- `GET /api/admin/users`: List all registered users.
- `POST /api/admin/users/{user_id}/credit-adjustments`: Add or subtract credits from a user's wallet.
- `GET /api/admin/providers` & `GET /api/admin/models`: Manage LLM providers and model configurations.
- `GET /api/admin/tools` & `POST /api/admin/tool-bindings`: Configure available tools and their scopes.
- `GET /api/admin/usage-events`: View global usage logs.
- `GET /api/admin/admin-actions`: Audit log of all administrative changes.

## 5. How to Connect & Use

### Connection Details
- **Base URL**: `http://<your-host>:8000/api`
- **Content-Type**: `application/json` (except for file uploads)

### Example: Check Admin Stats (curl)
```bash
curl -X GET "http://localhost:8000/api/admin/stats" \
     -H "Authorization: Bearer YOUR_ADMIN_PASSWORD"
```

### Example: Send a Message
```bash
curl -X POST "http://localhost:8000/api/chats/1/messages" \
     -H "Content-Type: application/json" \
     -d '{
       "content": "What is the status of my project?",
       "model_id": 1
     }'
```

Example response fragment:
```json
{
  "message": {
    "id": 42,
    "chat_id": 1,
    "role": "assistant",
    "content": "...",
    "reasoning_summary": "Reasoning covered: capabilities, pricing, constraints.",
    "reasoning_stages": [
      { "index": 1, "topic": "capabilities", "summary": "Model considered capabilities" },
      { "index": 2, "topic": "pricing", "summary": "Model considered pricing" }
    ]
  },
  "reasoning": {
    "available": true,
    "summary": "Reasoning covered: capabilities, pricing, constraints.",
    "stages": [
      { "index": 1, "topic": "capabilities", "summary": "Model considered capabilities" },
      { "index": 2, "topic": "pricing", "summary": "Model considered pricing" }
    ]
  }
}
```

## 6. Integration Notes
Since the removal of the dedicated web UI, the backend is the primary source of truth. 
- **API Clients**: Can build custom dashboards or interfaces using the endpoints listed above.
- **Telegram Bot**: Operates as a thin client, translating Telegram events into API calls (documented in `app/bot.py`).
- **Swagger Documentation**: Interactive documentation is available at `/docs` (Swagger UI) or `/redoc` for a detailed schema overview.
